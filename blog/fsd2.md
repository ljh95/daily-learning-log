## 1. '페이지 우선' 접근법: 추상화보다 응집력을 선택하다

FSD 2.1의 가장 큰 변화는 바로 "페이지 우선(pages first)" 접근법의 도입입니다. 이 새로운 방식은 간단명료합니다. 다른 페이지에서 재사용되지 않는 UI 블록, 폼, 데이터 로직 등은 이제 그것이 사용되는 특정 페이지의 슬라이스 내에 위치시켜야 합니다.

이러한 철학은 pages 레이어에만 국한되지 않습니다. widgets 레이어 역시 중요한 변화를 맞았습니다. 과거에 widgets이 단순히 여러 features와 entities를 조합하는 '껍데기' 역할에 가까웠다면, 이제는 위젯 외부에서 필요하지 않은 자체적인 상태 저장소, 비즈니스 로직, API 연동 코드까지 포함하는 독립적인 모듈로 격상되었습니다.

과거에는 재사용성을 최우선으로 고려하며 코드를 분리했다면, 이제는 특정 기능의 응집도를 우선하여 관련된 모든 코드를 한곳에 모으는 것을 권장합니다.

이 변화는 코드의 **응집도(cohesion)**를 극적으로 높여줍니다. 더 이상 단일 사용자 흐름을 수정하기 위해 features, entities, shared 등 여러 폴더를 넘나들 필요가 없습니다. 특정 페이지와 관련된 모든 코드가 한곳에 모여있어 개발 경험이 훨씬 직관적이고 효율적으로 바뀝니다.

## 2. 과거의 함정: '엔티티'와 '피처' 식별이 어려웠던 이유

FSD는 왜 이런 극적인 방향 전환을 했을까요? 놀랍게도, FSD는 과거 접근법의 문제점을 솔직하게 인정했습니다. FSD 2.0 방식에는 다음과 같은 주요 문제점들이 있었습니다.

- 낮은 코드 응집도: 단일 기능을 수정하기 위해 여러 폴더를 오가야만 했습니다. 이는 개발자의 인지 부하를 높이는 주요 원인이었습니다.
- 사용하지 않는 코드 삭제의 어려움: 기능이 여러 곳에 분산되어 있다 보니, 특정 기능이 더 이상 사용되지 않을 때 관련된 코드를 한 번에 찾아 삭제하기가 어려웠습니다.
- 높은 비즈니스 컨텍스트 요구: 엔티티와 피처 식별은 단순한 기술적 분리가 아니라, 비즈니스 도메인에 대한 깊은 이해를 요구하는 고급 기술이었습니다.
- 팀 내 해석의 불일치: 개발자마다 '엔티티'와 '피처'에 대한 정의가 달라 팀 내에서 혼란과 불필요한 논쟁을 유발했습니다.

FSD 문서에서는 이전 접근법의 어려움을 다음과 같이 명확하게 인정합니다.

Finding entities and features is still an advanced skill that needs to be developed over time... On the other hand, splitting by pages is natural and requires little training

하나의 아키텍처 방법론이 스스로의 단점을 공개적으로 인정하고 개선하는 것은 매우 드문 일입니다. 이는 FSD가 이론에만 갇히지 않고 실제 개발 현장의 목소리에 귀 기울이고 있음을 보여주는 중요한 대목입니다.

## 3. 사라진 치트시트: 때로는 없는 것이 더 낫다

또 하나의 놀라운 변화는 기존에 제공되던 '분해 치트시트(decomposition cheatsheet)'가 현재 목록에서 제외(unlisted)되었다는 사실입니다. 많은 개발자들이 FSD를 처음 접할 때 의지했던 이 가이드가 왜 사라졌을까요?

공식적인 이유는 충격적일 만큼 직설적입니다.

It proved to be more harmful than useful

이 결정은 FSD 방법론의 방향성을 명확히 보여줍니다. 엄격하고 고정된 규칙을 기계적으로 따르기보다, 개발자가 상황에 맞게 실용적이고 유연한 판단을 내리는 것을 더 중시하게 된 것입니다. 치트시트의 부재는 오히려 개발자들에게 더 깊은 고민을 유도하고, 맹목적인 규칙 적용으로 인한 부작용을 막는 긍정적인 효과를 가져올 수 있습니다. 이는 방법론의 겸손하고 실용적인 발전을 보여주는 훌륭한 사례입니다.

## 결론

Feature-Sliced Design의 진화는 명확한 방향을 가리키고 있습니다. 맹목적인 추상화와 코드 분리보다는 개발자 경험, 코드 응집도, 그리고 단순함을 우선시하는 실용적인 길로 나아가고 있습니다. '페이지 우선' 접근법과 치트시트의 제거는 이러한 철학적 변화를 상징하는 가장 강력한 증거입니다.

사실 이러한 변화는 FSD에만 국한된 움직임이 아닙니다. Remix의 라우트 기반 데이터 로딩이나 서버 컴포넌트의 부상처럼, 프론트엔드 생태계 전반에서 과도한 추상화를 지양하고 코드의 응집도와 배치(co-location)를 중시하는 실용적인 흐름이 나타나고 있습니다. FSD의 진화는 바로 이 거대한 흐름의 일부입니다.
